오늘은 웹이 어떻게 세 겹으로 나뉘는지, 그 뒤의 컴퓨터 원리(프로세스·스레드, 동기·비동기)까지 배우고
직접 브라우저 미니게임을 만들어봤다. 배운 걸 정리하고, 만든 게임도 소개한다.

## 1. 웹의 세 겹 — HTML · CSS · JavaScript

웹은 처음부터 세 겹으로 설계된 게 아니라, 30년에 걸쳐 문제를 하나씩 풀며 지금 모양이 됐다.

- **HTML** — 1989년 CERN의 팀 버너스리가 "흩어진 문서를 단어에서 단어로 건너뛰며 연결하자"(하이퍼텍스트)는 제안에서 출발. 웹의 **뼈대**.
- **CSS** — 내용과 꾸밈이 붙어 있으면 수정이 여기저기 번지니, 꾸밈을 분리. 웹의 **옷/화장**.
- **JavaScript** — 1995년 넷스케이프에서 브렌던 아이크가 **열흘 만에** 만든 언어. 첫 임무는 "폼 검증"(제출 전 이메일 형식 확인). 웹의 **근육/움직임**.

> 자바(Java)와 자바스크립트(JavaScript)는 이름만 닮은 남남 — "햄과 햄스터의 관계"라는 농담이 있다.
{: .prompt-tip }

10년 뒤 2005년 구글 지도가 "새로고침 없이 지도를 끌면서 데이터를 받아오는"(AJAX) 방식을 보여주며,
웹은 '문서'에서 '애플리케이션'으로 넘어갔다.

| 겹 | 태어난 문제 | 검토 기준 |
|---|---|---|
| HTML | 흩어진 정보를 의미 있게 연결 | 정보·용어가 빠지지 않았는가 |
| CSS | 내용과 꾸밈이 붙어 수정이 번짐 | 읽히는가, 깨지는 곳은 없는가 |
| JavaScript | 눌러도 반응 없는 정적 문서 | 행동 뒤의 변화가 기대대로인가 |

**AI에게 지시할 때도 겹을 지정하면 좋다.** "페이지 고쳐줘"(범위 불명) 대신
"카드 내용은 그대로 두고 모바일 간격만 조정해줘"(겹 지정) → AI가 손댈 범위가 좁아지고, 멀쩡한 부분이 덜 흔들린다.

## 2. 프론트엔드 / 백엔드

| 구분 | 프론트엔드 | 백엔드 |
|---|---|---|
| 위치 | 브라우저 화면, 버튼, 입력창 | 서버, API, 데이터 저장소, 권한 |
| 식당 비유 | 홀·메뉴판·주문 화면 | 주방·창고·주문 장부 |
| 핵심 질문 | 무엇을 보고/누르고/입력하는가 | 데이터가 어디 저장되고 누가 다시 보는가 |

## 3. CS 교양 — 프로세스·스레드, 동기·비동기

**프로세스 vs 스레드**
- **프로세스** = 실행 중인 프로그램 하나. **독립된 메모리**(격리). 하나가 죽어도 다른 건 멀쩡.
- **스레드** = 프로세스 안의 일꾼. 메모리를 **공유**. 가볍지만, 하나가 터지면 전체가 위험.
- 커널은 **메모리는 프로세스 단위로, CPU 시간은 스레드 단위로** 나눠준다.

**동기 vs 비동기** — 전화 vs 문자
- **동기**: 전화. 끝날 때까지 붙잡혀 있음(다른 일 못 함).
- **비동기**: 문자/카페 진동벨. 걸어두고 다른 일 하다가, 신호 오면 처리.

**멀티스레딩 vs 멀티프로세싱**

| 기준 | 멀티스레딩 | 멀티프로세싱 |
|---|---|---|
| 메모리 | 공유 | 격리 |
| 적합한 일 | I/O 대기 많을 때(네트워크·파일) | CPU 계산 무거울 때(연산·인코딩) |
| 안정성 | 하나 터지면 전체 위험 | 하나 죽어도 나머지 멀쩡 |

> 파이썬 실전 포인트: GIL 때문에 멀티스레딩으론 CPU 작업이 진짜 병렬이 안 된다.
> → CPU 무거운 일은 `multiprocessing`, 네트워크·파일 대기 일은 `threading`/`asyncio`.

## 4. 바이브 코딩 루프 — 계획 → 만들기 → 다듬기

전문 개발팀은 코드 전에 **PRD(제품 요구사항 문서)**를 쓴다. 바이브 코딩에선 이게 더 중요하다 —
AI는 내 마지막 한마디에 맞춰 움직여서, 목표가 글로 고정 안 되면 대화가 길어질수록 방향이 샌다.
그래서 **"먼저 명세를 쓰고 AI에게 시킨다"(스펙 주도 개발)**. 거창할 필요 없이 15~30분, 한 장이면 충분하다.

- 무엇을 / 누가 / 규칙과 끝나는 조건 / 화면 요소 / 예외 상황

## 5. 오늘 만든 미니게임 🎮 — 페어 프로그래밍으로

배운 세 겹을 실제로 써서, 브라우저에서 바로 플레이되는 게임을 만들었다.
가장 크게 배운 방식은 **페어 프로그래밍**이다. 조원과 짝을 이뤄
**드라이버(키보드 잡고 코딩)**와 **네비게이터(방향 제시·타임키퍼)** 역할을 **10분마다 번갈아** 맡았다.

**먼저 혼자 연습 (사전 준비)**

- 😤 **[꼰대 상사 당황시키기](https://nike1137-svg.github.io/kkondae-game/)** — 잔소리하는 상사를 연타로 제압하는 5판 승부. 페어 전에 혼자 만들어본 연습작.

**내가 드라이버 (내 GitHub)**

- 🐭 **[찍찍이 잡기](https://nike1137-svg.github.io/mole-game/)** — 30초 안에 움직이는 찍찍이를 잡는 두더지잡기형
- ⚫ **[오목](https://nike1137-svg.github.io/game/omok/)** — 흑백 번갈아 두는 전통 보드게임

👉 위 3종 모음(대문): [nike1137-svg.github.io/game](https://nike1137-svg.github.io/game/)

**내가 네비게이터 (조원이 드라이버)**

- 🎯 **[KHGAME](https://cjwmanelf.github.io/KHGAME/)** — 조원이 드라이빙, 내가 방향을 잡은 게임
- 🐰 **[토끼잡기](https://cjwmanelf.github.io/KHGAME2/)** — 움직이는 토끼들을 잡는 5판 승부

각각 `index.html` / `styles.css` / `script.js` 세 파일로 나눠 만들었다.
규칙(끝나는 조건, 예외 상황)을 먼저 정하고 AI에게 시키니, 결과가 이상해도 무엇이 잘못인지 판정할 기준이 생겼다.
그리고 드라이버·네비게이터를 번갈아 하니, 내가 직접 안 친 코드도 흐름을 이해하게 됐다.
