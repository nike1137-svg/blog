지난 며칠은 병원동행 앱 백엔드를 붙잡고 있었다. 이번 Main Quest는 결이 다르다 — 기획부터 배포까지, 하나의 서비스를 처음부터 끝까지 내 손으로 굴려보는 퀘스트였다. 다 끝내고 보니 깨달은 게 하나 있다. **혼자 다 만든 게 아니라, 곳곳에서 잘 빌려 왔다.** AI 기능 자체는 다른 프로젝트에서 돌던 에이전트(로고스)를 그대로 가져왔고, 그걸 배포된 앱까지 연결하는 방식(터널로 노출하고 PC가 켜져 있을 때만 동작)은 예전에 만든 새록이의 운영 패턴을 그대로 재활용했다. 디자인은 팀원의 눈을 빌렸다. 엔진은 여기저기서 빌려왔지만, 방향키는 내가 잡았다.

## 1. 한 문장에서 시작했다

거창하게 안 갔다. 딱 한 줄로 정했다.

> "돌봄제공자가 어르신의 복약을 기록하고, 교대자·보호자와 공유하는 웹앱"

시니어케어 3년 하면서 매일 본 장면이다. 인지저하·거동불편 어르신은 약을 먹었는지 스스로 기억하지도, 챙기지도 못한다. 교대할 때 "아까 드셨어요?"가 구두로만 오간다. 그 pain을 그대로 문제 정의로 옮겼다.

> **📌 알고 가기 — PRD가 뭔가**
> 코드 짜기 전에 "무엇을·왜 만드는지"를 합의하는 문서. 짜다 보면 자꾸 기능이 늘어나는데, PRD가 범위(scope)를 고정해준다. MVP는 자랑할 기능 목록이 아니라 "이거 없으면 서비스가 성립 안 되는 최소한"이라는 걸 이번에 처음 체감했다.

## 2. 데이터부터, 화면은 나중

화면부터 만들면 "이 데이터 어디서 오지?"에서 무너진다. 그래서 순서를 `데이터 → API → 화면`으로 잡았다. 설계 교훈 두 개.

**하나 — 저장할 것과 계산할 것을 나눈다.** "오늘의 체크리스트"는 저장하지 않는다. 등록된 약 × 오늘 날짜로 매번 계산하고, 실제로 체크한 것만 기록에 남긴다.

**둘 — 단순하게 잡고, 필요할 때 늘린다.** 복용 시간대를 컬럼으로 쪼개지 않고 배열 하나로 뒀다. 나중에 "영양제는 시간 무관, 하루 한 번"이라는 케이스가 튀어나왔는데, 배열이라 값 하나(`시간무관`)만 추가하면 끝났다. 단순한 설계가 실제로 이득을 돌려준 순간.

## 3. curl로 먼저, 눈으로 나중

화면 만들기 전에 API부터 curl로 두드려봤다. 로그인 붙인 뒤 터진 **401 에러**는 추측 대신 서버에 로그를 심어 직접 원인을 봤다 — 토큰 검증용 클라이언트와 조회용 클라이언트가 내부에서 충돌하고 있었다. 둘을 분리하니 풀렸다. "왜 안 되는지 로그로 좁히기" — 이게 디버깅의 정석이라는 걸 몸으로 했다.

> **📌 알고 가기 — RLS(Row-Level Security)**
> DB 자체에 "이 행은 주인한테만 보여줘"라는 규칙을 새기는 것. 도서관에서 내 대출 카드로는 내 기록만 조회되는 것과 같다.

## 4. 붙이기 전에 멈춰서 물었다

기존 병원동행 프로젝트의 Supabase를 재사용하려다가, 붙이기 전에 "두 앱이 한 DB를 쓰면 뭐가 터지지?"를 먼저 따졌다. 건강정보인데 공개 키를 공유하면 데이터 격리가 안 된다는 결론이 나와서, 새 프로젝트로 **분리**했다. 로그인을 붙일 때도 개인정보 제약부터 다 확인하고 가자고 먼저 브레이크를 걸었다.

AI 기능(요약·자연어 입력)도 처음엔 넣으려 했지만 자원 제약(무료 크레딧 소진, 카드 필요)으로 **안 넣기로 판단**했다. 로그인+RLS가 이미 "새로운 시도" 요건을 채웠기 때문이다. 한정된 시간·돈에서 "이건 지금 투자 가치가 낮다"를 계산하고 멈추는 것도 결정이다 — 이 판단은 나중에 뒤집힌다.

## 5. 배포, 그리고 GitHub Pages를 스스로 의심했다

Vercel로 배포하고 제출까지 마쳤다. 그런데 채점자가 "GitHub Pages로 해달라"고 했을 때, 곧바로 맞추려 하지 않고 먼저 물었다. GitHub Pages는 **미리 구운 빵을 진열하는 곳**(정적 파일만)이고, 우리 앱은 **주문 시 즉석 조리하는 식당**(요청마다 서버가 API 실행)이다. 서버 코드를 못 돌리니 구조가 안 맞았다. 과제 원문이 인정하는 Vercel로 그대로 갔다.

## 6. 팀원 피드백으로 다시 손을 댔다

제출 끝났다고 끝난 게 아니었다. 디자인 전공 팀원이 화면을 보고 지적했다.

- "이메일 발송 제한 에러가 영어 원문 그대로 뜨는데, 실사용자는 못 알아들을 거야" → 에러를 한국어로 번역하고, 평소에도 보이는 안내 문구를 추가했다.
- "로그인 화면만 딱 나오면 이게 뭐 하는 서비스인지 첫 방문자는 모르잖아" → 로그인 위에 서비스 소개(랜딩) 섹션을 얹었다.

혼자 만들 때는 안 보이던 구멍이 다른 사람 눈에는 바로 보인다는 걸 다시 배웠다.

## 7. 빌려온 AI — 로고스를 다시 꺼냈다

AI를 접었던 이유는 비용이었다. 그런데 PC에 이미 **로고스**라는 게 있었다 — CrewAI로 만든 개인용 에이전트, LLM은 Vertex AI(Gemini)를 무료 크레딧으로 돌리는 중이었다. 다른 목적으로 만든 걸 다시 뜯어봤다.

> **📌 알고 가기 — CrewAI**
> 여러 AI 에이전트를 역할별로 만들어 한 팀처럼 협업시키는 프레임워크. 단일 챗봇이 아니라 "각자 맡은 역할이 있는 팀"에 가깝다.

**연동 구조**: 배포된 앱 → 로그인 인증이 걸린 API 라우트 → Cloudflare 터널 → PC(WSL)의 Flask 서버 → CrewAI 에이전트 → Vertex AI. 배포 사이트에서 클릭하면 실제로 내 컴퓨터까지 왕복해서 응답이 온다.

**기능은 안전하게 좁혔다.** "약 이름 넣으면 AI가 복용 주의사항을 생성"은 하지 않았다 — 새로운 의료정보를 만드는 거라 환각(잘못된 정보) 위험이 있다. 대신 "매일 아침저녁 혈압약 한 알" 같은 문장을 **구조화만** 하는 자연어 자동완성으로 갔다. 새 정보를 만들지 않고 사용자가 쓴 걸 정리만 하니 위험이 훨씬 적다. 서버 쪽에는 `time_slots`가 정해진 5개 값 밖으로 못 나가게 검증도 걸어뒀다.

**사용량도 관리 대상이었다.** 클로드 사용 한도가 거의 다 찬 시점에, 지금 배포돼 있는 앱은 손대지 않고 **로고스 쪽(내 PC) 준비만 먼저 끝내는** 순서로 작업을 쪼갰다. 한도가 끊겨도 이미 제출한 배포는 항상 안전한 상태로 남도록. 리스크를 분리하는 것도 판단이다.

이 구조, 어디서 본 것 같았다. 예전에 만든 새록이도 똑같이 **Cloudflare 터널로 PC를 세상과 잇고, PC가 켜져 있을 때만 서비스가 돈다.** 그때 겪은 한계(터널 재시작마다 주소가 바뀜, PC를 껐다 켤 때마다 재실행 필요)를 이번에도 그대로 감수하기로 했다. 대신 AI가 꺼져 있어도 등록·체크·로그인 같은 핵심 기능은 항상 정상 동작하도록 설계해서, 이 기능이 죽어도 서비스 전체가 죽지는 않는다.

## 8. "개발자가 만든 것"에서 "병원 서비스처럼"

팀원이 다른 조 과제를 보고 준 피드백을 참고해, 화면을 다시 설계했다.

- **상태색**: 시간대별로 지금이 몇 시인지 봐서 완료(초록)·예정(파랑)·놓침(빨강)·시간무관(민트)을 자동으로 갈랐다.
- **접근성**: 버튼·입력창 높이를 52~56px로, 본문 글자를 16px로 키웠다. 대상이 고령층이면 이건 선택이 아니라 필수다.
- **온보딩**: 처음 로그인해서 약이 하나도 없을 때, "약 등록 → 목록 표시 → 체크 시작" 3단계를 미리 보여줬다.
- **PWA**: 매니페스트와 아이콘을 넣어 휴대폰 홈 화면에 설치 가능하게 만들었다. 가벼운 버전이라 완벽한 오프라인 지원은 아니지만, "앱처럼 보이는" 첫 인상은 확보했다.

랜딩화면 기능 카드 순서도 논의했다. "AI 자동완성"을 로그인 폼 바로 위, 맨 마지막 카드로 두기로 했다 — 기본 신뢰(체크·보안)를 먼저 보여주고, 로그인을 누르기 직전에 "이런 것도 되네"로 마무리 짓는 흐름이다.

## 내 역할

| 클로드/도구가 한 일 | 내가 판단한 일 |
| --- | --- |
| API·컴포넌트·SQL 타이핑, 문법·설정 | 무엇을 만들지(주제), 스택 분리 결정 |
| 에러 원인 후보 제시, 코드 디버깅 | 개인정보 점검 먼저 하자는 브레이크 |
| 정적/서버 배포 차이 설명 | AI를 뺄지, 로고스로 되살릴지 결정 |
| 디자인 시스템 초안 제시 | 카드 순서·문구를 실제 사용자 흐름으로 확정 |
| 사용량 리스크 분리 제안 | 어느 작업을 먼저·나중에 할지 순서 확정 |

가장 효과가 컸던 프롬프트 하나를 꼽자면 이거였다.

> "기존 것과 새로 만들 것을 같이 쓰면서 생기는 오류·문제점 파악이 먼저였어. 이중으로 확인하고, 내가 승인하면 그때 작업 진행해."

이 한 줄이 데이터 격리 사고를 미리 막았다. 도구에 실행은 맡기되, 검토와 승인 책임은 내가 쥔다 — 방향키를 잡는다는 게 이런 거였다.

GitHub Contributors에 내 이름만 떠 있는 걸 보고 잠깐 헷갈렸다. "이거 내가 다 한 건가?" 아니다 — 코드는 도구와 함께 썼다. 하지만 **무엇을 만들지, 이게 맞는지, 언제 멈추고 언제 다시 손댈지는 내가 정했다.** 코드를 누가 타이핑했느냐와 이 작업이 누구 것이냐는 다른 질문이다.

**다음 한 걸음**: 로고스 터널을 매번 재발급하지 않아도 되는 고정 주소로 바꾸거나, 이메일 발송 제한 자체를 커스텀 SMTP로 풀어보기.
