## 1. 프레임워크란? — '뼈대/틀'

집을 지을 때 기둥·배관·전선을 0부터 설계하지 않고 **검증된 골조** 위에 방 배치를 올린다.
웹 프레임워크도 같다. 버튼·카드 재사용, 데이터가 바뀌면 화면도 바꾸기, 주소에 따라 다른 화면 보여주기처럼
**웹에서 반복되는 일을 미리 정리해 둔 개발 골조**다.

**라이브러리 vs 프레임워크 — 제어의 역전(IoC)**
- 라이브러리 = *부품 상자*. 내가 필요할 때 꺼내 쓴다.
- 프레임워크 = *골조 + 약속*. 전체 흐름을 쥐고 있다가, 정해진 순간에 **내 코드를 불러 준다**.
- 이 뒤집힘을 **제어의 역전(Inversion of Control)** 이라 부른다.

## 2. CS 교양 — 추상화와 모듈화

프레임워크가 반복을 덜어 주는 뿌리엔 두 개념이 있다.

- **추상화(Abstraction)**: 복잡한 속을 감추고 꼭 필요한 조작만 밖으로. *자동차 운전 시 엔진 연소를 몰라도 페달·핸들만 알면 됨.* 웹에선 손 많이 가는 DOM 조작을 감추고, 개발자는 "상태가 이러면 화면은 이래야 한다"만 적는 **선언형(declarative) UI**.
- **모듈화(Modularity)**: 큰 덩어리를 갈아 끼울 수 있는 조각으로. *레고 블록.* 화면을 **컴포넌트**로 나누면, 그 조각만 고치거나 다른 화면에 옮길 수 있다.

> 역사 한 걸음: 1972년 데이비드 파나스는 "관련된 것끼리가 아니라 **바뀌어야 할 이유가 같은 것끼리** 모아라"고 했다. 50년 뒤 React·Vue·Svelte에도 그대로 살아 있다.
{: .prompt-info }

**바이브 코딩과의 연결**: 문법은 AI가 채운다. 하지만 *무엇을 조각으로 나누고(모듈화), 무엇을 상태로 둘지(추상화)* 는 내가 판단한다. 이 판단이 문법보다 오래 쓸모 있다.

## 3. DOM과 가상 DOM

- **DOM(Document Object Model)**: 웹페이지를 브라우저가 다룰 수 있게 **객체(노드)들의 나무 구조**로 표현한 것. 화면의 제목·버튼·목록 하나하나가 노드. 즉 DOM은 "내가 명령으로 주무르는 **대상(재료)**".
- **가상 DOM**: 메모리 위의 가벼운 JavaScript 객체. 실제 화면이 아니라 *"화면이 이렇게 생겨야 한다"고 적어둔 설계도(메모)*.

## 4. 프레임워크 비교 — 하나만 정답이 아닌 이유

| 도구 | 스스로를 이렇게 설명 |
|---|---|
| React | UI를 렌더링하는 JS **라이브러리**. 화면을 컴포넌트로 나누는 사고를 퍼뜨림 |
| Vue | UI를 만드는 JS **프레임워크**. 점진적 도입 강조 |
| Angular | Google 유지. 라우팅·폼·업데이트 묶음이 강해 큰 조직 장기 운영에 |
| Svelte | **컴파일러**로 브라우저가 할 일을 줄임 |
| Solid | 반응형 UI **성능**을 극한까지 |
| Qwik | 서버 렌더링을 브라우저가 이어받는 **resumability** |

**문법은 AI가 짠다 → 그래서 '무엇을 고르냐'가 더 중요**
- AI는 **자료 많은 프레임워크를 더 정확히** 짠다. (DesignBench 벤치마크: React·Vue 컴파일 성공률 0.83+, Angular 0.69~0.76 / arXiv:2506.06251, 2025)
- **규칙이 일관된** 프레임워크일수록 AI가 매번 비슷한 구조로 낸다.
- 가장 많이 쓰는 건 React(Stack Overflow 2025, 44.7% 1위). 하지만 **만족도**는 Svelte가 최고(~74%), React는 62%로 하락(State of JS 2024).
- 정리: *자료·AI 지원 최우선 → React 계열*, *간결함·개발 경험 중시 → Svelte* 도 합리적. **"많이 쓴다 ≠ 내 조건에 맞다".**

> 확인법: 후보에게 같은 요청("상품 목록 화면 만들어 줘")을 세 번 시켜, 구조가 매번 비슷하면 AI 궁합이 좋은 것.
{: .prompt-tip }

## 5. REST API의 네 가지 동사

프론트엔드가 백엔드에 URL로 데이터를 요청하는 방식. HTTP 동사를 데이터 조작에 대응시킨다.

| 동사 | URL 예 | 식당 비유 | 의미 |
|---|---|---|---|
| GET | `/api/experts` | "메뉴판 보여줘" | 목록 가져오기 |
| POST | `/api/experts` | "이거 주문할게" | 새 데이터 생성 |
| PUT | `/api/experts/1` | "1번 이렇게 바꿔줘" | 기존 데이터 수정 |
| DELETE | `/api/experts/1` | "1번 취소" | 데이터 삭제 |

(GET·POST·PUT·DELETE ↔ 어제 배운 **CRUD**의 Read·Create·Update·Delete와 그대로 대응된다.)

## 6. 목업 데이터(Mock Data) — 가짜지만 진짜처럼

진짜 API가 보내줄 것과 **같은 구조의 가짜 데이터**를 코드에 직접 적어두는 것. 백엔드가 없어도 화면이 진짜처럼 동작한다.

- **백엔드를 안 기다림** → 프론트·백엔드 팀이 동시에 일하는 *병렬 개발*
- **디자인·인터랙션 먼저 검증**
- **AI 코딩에서 강력** — "이 데이터로 화면 만들어줘" 하면 백엔드 없이 완성 화면 즉시 확인
- **나중에 진짜 API로 교체 쉬움** — 구조만 맞춰두면 거의 한 줄 교체

---

한 줄 정리: **프레임워크는 문법 묶음이 아니라, 복잡함을 감추고(추상화) 조각으로 나눠(모듈화) 반복을 줄이는 도구다.** 문법은 AI가 짜니, 나는 *무엇을 고르고 어떻게 나눌지* 를 판단하면 된다.
