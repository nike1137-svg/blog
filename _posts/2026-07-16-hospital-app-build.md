**화면(프론트엔드)부터 직접 만들고**, 거기에 **진짜 백엔드를 붙이고**, **인터넷에 배포**까지 한 바퀴 다 돌아봤다. 홀만 있던 가게에 드디어 주방이 생긴 느낌이다. 그 과정을 정리해 둔다.

> ⚠️ 보안상 API 키·접속 정보 같은 비밀값은 이 글에 일절 담지 않았다. 과정과 배운 점만 기록한다.

## 만든 것

- **병원동행 매칭 서비스** — 병원 갈 때 필요한 어르신·보호자와 요양보호사를 잇는 앱
- 핵심 기능: 요양보호사 등록 · 동행 요청 · 조건별 매칭 목록
- 스택: **Next.js**(프론트) + **Supabase**(백엔드/DB) + **GitHub Pages**(배포)

## 1. 프론트엔드 — 화면부터 직접 만들기

백엔드보다 **화면(프론트엔드)을 먼저** 만들었다. 가짜 데이터(JSON) 6명으로 돌려서, 서버 없이도 흐름을 눈으로 다 확인할 수 있게. "고르는 경험"을 먼저 완성해 두는 게 핵심이었다.

만든 화면은 5개다.

- **홈** — 서비스 소개 + 이용 흐름 3단계
- **요양보호사 찾기(매칭 목록)** — 지역·경력·자격으로 거르고 정렬
- **요양보호사 상세** — 개별 프로필
- **요양보호사 등록** / **동행 요청** — 입력 폼 2종

만들면서 신경 쓴 것들:

- **컴포넌트로 쪼개기**: 카드, 목록, 내비게이션을 각각 부품으로 나눴다. 예전에 정리했던 *[파일 3개 분리 = 관심사의 분리]* 원칙이, 화면이 커지니 자연스럽게 "컴포넌트 분리"로 이어졌다. 한 부품만 고치면 되니 수정이 편했다.
- **필터·정렬 UI**: 지역 선택·경력순 정렬·"자격 보유자만" 체크박스를 붙였다. 사용자가 조작할 때 화면이 즉시 바뀌어야 해서, 이 부분은 브라우저에서 도는 조각(클라이언트 컴포넌트)으로 만들었다.
- **반응형**: Tailwind로 폰에서도, PC에서도 깨지지 않게. 어르신·보호자가 많이 쓸 서비스라 모바일을 신경 썼다.

이렇게 목업으로 화면을 탄탄히 만들어 두니, 뒤에 백엔드는 정말 "한 겹 덧입히기"로 끝났다.

## 2. 백엔드 붙이기 — Supabase

서버를 직접 세우는 대신, DB·인증·저장을 빌려 쓰는 Supabase를 골랐다. 테이블 2개(요양보호사 / 동행요청)를 만들고, 목업 데이터를 DB로 옮겼다.

가장 인상 깊었던 건 **RLS(행 수준 보안)**. Supabase는 브라우저가 공개 키로 DB에 직접 접근하는 구조라, RLS를 켜고 규칙을 짜야만 안전하다. 안 켜면 그 공개 키로 누구나 데이터를 읽고 고치고 지울 수 있다. 실제로 켠 뒤에 "조회는 되는데 삭제·수정은 막히는지" 직접 테스트해서 확인했다.

## 3. 가짜 → 진짜로 바꾸기

여기서 신기했던 점. 화면이 **"가짜 데이터를 불러오던 한 줄"을 "진짜 서버에 요청하는 한 줄"로 바꾸기만** 하면 됐다. 목업으로 화면을 잘 만들어 두니, 백엔드는 정말 "한 겹 덧입히기"였다.

대신 모든 서버 호출을 **로딩 / 성공 / 에러** 세 상태로 나눠 화면을 준비했다. 데이터 오기 전 깜빡임, 서버 오류 시 멈춘 듯 보이는 문제를 막기 위해서다.

## 4. 배포 — GitHub Pages

GitHub Pages는 서버 없이 정적 파일만 올리는 곳이라, Next.js를 **정적 파일로 뽑아내는(static export)** 설정을 추가했다. 주소가 `/저장소이름/` 하위라 경로(basePath)도 맞췄다.

처음엔 수동으로 결과물을 올렸다가, 나중에 **GitHub Actions 자동배포**로 바꿨다. 이제 코드를 고치고 `push`만 하면 자동으로 빌드·배포된다 — 블로그와 똑같은 방식이다.

## 5. 보안에서 배운 것

가장 많이 되뇐 원칙:

- **코드 = 공개해도 되는 것 / 환경변수 = 절대 공개하면 안 되는 것**
- 키는 코드에 직접 안 쓰고 환경변수로, `.env`는 깃에 안 올라가게.
- **주의**: 프론트에 들어가는 값(`NEXT_PUBLIC_` 접두사)은 브라우저에 그대로 노출된다. 그래서 **공개돼도 되는 키만** 프론트에 쓰고, 모든 걸 무시하는 마스터 키는 프론트에 절대 넣지 않는다.
- 진짜 보호는 결국 **RLS**가 한다.

## 6. 막혔던 순간들

- **계약 불일치**: 프론트가 기대하는 데이터 모양과 백엔드가 주는 모양이 어긋나면 화면에 `undefined`가 뜬다. → 주고받을 JSON 모양을 **API 명세 문서로 먼저 적어 두고** 그 기준으로 양쪽을 짰다.
- **정적 배포에서 상세 페이지 404**: 새로 등록된 항목의 상세 페이지가 안 열렸다. → 경로 방식 대신 **쿼리 방식**으로 바꿔 어떤 항목이든 열리게 했다.
- **배포 권한**: 자동배포용 워크플로 파일을 올리려면 별도 권한이 필요해서, 한 번 인증을 추가했다.

## 마무리

지난 시간에 배운 네트워크·보안·배포가, 목업 화면에 백엔드라는 '한 겹'을 안전하게 세상에 열어 두기 위한 준비였다는 걸 이제 알겠다. **친구가 링크를 열 수 있고, 새로고침해도 데이터가 남는 것** — 그게 이 과정의 끝에서 실제로 이뤄졌다.

다음엔 로그인·권한, 그리고 매칭 로직을 더 다듬어 보고 싶다.
