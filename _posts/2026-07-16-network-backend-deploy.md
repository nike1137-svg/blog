오늘 학습 본문. 전자통신·네트워크 배경과 이어지는 IPv6부터, 요청이 서버에 닿아 처리되고, 만든 걸 배포하는 데까지 한 흐름으로 정리한다.

## 1. IPv6 — 왜, 어떻게

- **왜**: IPv4는 32비트라 약 43억 개뿐 → 스마트폰·IoT·클라우드로 고갈. 임시방편이 **NAT**였지만 P2P·포트포워딩·End-to-End를 복잡하게 만든다. IPv6는 **128비트**로 확장해 근본 해결(약 3.4×10³⁸개).
- **구조**: 16진수 4자리 × 8그룹, 콜론(`:`) 구분. 핵심은 **상위 64비트=네트워크 / 하위 64비트=인터페이스**로 반씩 나뉜다는 것. 서브넷 하나에 2⁶⁴개라 사실상 무한.
- **축약**: 각 그룹 앞의 `0` 생략(`0db8`→`db8`), 연속된 `0` 그룹은 `::`로 한 번만 압축. `2001:0db8:0000:...:0001` → `2001:db8::1`. (`::`는 주소당 딱 한 번만)
- **주소 종류**: Global Unicast(`2000::/3`, 공인) / **Link-Local(`fe80::/10`)** — 인터페이스가 켜지면 무조건 자동 생성, 로컬 통신 항상 가능 / ULA(`fd00::/8`, 사설, IPv4 192.168에 해당) / Multicast(`ff00::/8`). **브로드캐스트는 사라지고 멀티캐스트로 대체**(모든 노드 `ff02::1`).
- **운용 관점**: NAT 원칙적으로 불필요(단 방화벽은 여전히 필요), ARP→**NDP**로 교체, DHCP 없이 **SLAAC**로 주소 자동 설정. *Tailscale의 `fd7a::` 주소가 바로 ULA 기반 IPv6다.*

## 2. TTL — 홉 제한 vs 캐시 시간

- **IP 패킷 TTL** = 라우터를 지날 때마다 1씩 감소, 0이면 폐기 → **무한 루프 방지**. IPv6에선 이름이 **Hop Limit**으로 바뀜(기능 동일).
- **활용**: `ping` 응답의 TTL로 OS·거리 추측(Linux 64, Windows 128 시작). `traceroute`/`tracert`는 TTL을 1→2→3…으로 늘려 경로상 라우터를 순서대로 알아냄.
- **DNS·캐시 TTL** = "이 정보를 몇 초 캐시할지"(초 단위 시간). Cloudflare A/CNAME 레코드의 TTL이 이것 — IP 곧 바꿀 거면 짧게(60초), 안정적이면 길게.
- **핵심 구분**: IP 계층 TTL = **홉 수**, DNS·캐시 TTL = **초 단위 시간**.

> 내 IP 확인: 🐧 `ip addr show`(사설) / `curl ifconfig.me`(공인), 🪟 `ipconfig` / `(Invoke-WebRequest ifconfig.me/ip).Content.Trim()`
{: .prompt-tip }

## 3. 내부망 / 외부망 — "왜 내 링크가 밖에선 안 보이지?"

- **내부망(LAN)**: 공유기로 묶인 '우리끼리' 네트워크. 사설 IP(`192.168.x.x`).
- **외부망(WAN=인터넷)**: 밖에서 우리 집은 **공인 IP 하나**로만 보임.
- 문지기가 **공유기(NAT)**. 안→밖은 자유, **밖→안은 기본 차단**(그래서 평소 안전). `localhost:3000`이 나한테만 보이는 이유 = '내부망의 내 컴퓨터'에만 있으니까.

## 4. 포트와 방화벽

- IP가 '건물 주소'면 **포트는 '호실 번호'**(0~65535). 웹은 보통 `80`(HTTP)·`443`(HTTPS), 개발 중엔 `localhost:3000`.
- **방화벽** = 그 문을 지키는 경비. 필요 없는 문은 닫아 둔다.

## 5. HTTP / HTTPS와 자물쇠의 원리

- **HTTP**: "이 페이지 줘"(요청) → "여기 있어, 200 OK"(응답)의 규칙.
- **HTTPS**: 그 대화를 암호로 감싼 것. **엽서 vs 밀봉 봉투**. 로그인·결제엔 필수.
- **공개키 방식**: 공개키 = *누구나 넣을 수 있는 우편함 투입구*, 개인키 = *서버만 가진 여는 열쇠*. 실제론 느려서, 접속 초반에 공개키로 '임시 열쇠(세션 키)'를 안전하게 나눈 뒤 빠른 방식으로 대화 → 이 첫 인사가 **TLS 핸드셰이크**.

**요청 한 번의 여정** (`mysite.com` 엔터 후 1초):
1. DNS에 "주소(IP) 어디?" 질의 → 2. 그 IP의 `443` 문에 HTTPS로 연결 → 3. HTTP 요청 전송, 서버가 응답 → 4. 브라우저가 화면으로 렌더. (택배: 주소 검색 → 이동 → 수령 → 개봉)

## 6. 백엔드 — "프론트는 못 믿는다"

백엔드는 한 문장: **요청을 받아서, 처리하고, 응답을 돌려준다.** 홀(프론트)에서 주문서(요청)를 받고 → 레시피(비즈니스 로직)대로 조리 → 창고(DB)에서 재료를 꺼내 → 완성품(응답)을 내보낸다.

> **가장 중요한 보안 원칙**: 가격·수량·권한 같은 검증은 **반드시 백엔드에서 다시** 한다. 프론트는 사용자 컴퓨터에서 도니 얼마든지 조작 가능하다. 화면에 "10,000원"이어도 사용자가 요청을 100원으로 바꿔 보낼 수 있으므로, 최종 가격은 서버가 자기 데이터로 다시 계산해야 한다. (출처: OWASP)
{: .prompt-warning }

- 백엔드 언어/프레임워크: JS/TS→Express·NestJS, Python→Django·Flask·FastAPI, 그 외 Rails·Go·Java/Spring.

## 7. 환경변수 — 비밀은 코드에 적지 않는다

API 키를 코드에 박아 GitHub에 올리면 몇 분 만에 봇이 훔쳐 간다. (GitGuardian 집계: 2025년 공개 GitHub 유출 비밀 약 2,865만 개, 전년比 +34%, AI 키 유출 +81% — "State of Secrets Sprawl 2026")

- 나쁜 예: `apiKey = "sk-a1b2c3..."` (그대로 유출)
- 좋은 예: `apiKey = 환경변수에서 불러오기` + 실제 값은 `.env`에 보관 + `.gitignore`로 깃 차단
- AI에게: **"API 키·DB 접속 정보는 코드에 넣지 말고 환경변수로 빼줘. `.env`는 깃에 안 올라가게 해줘."**

## 8. 배포 — 초보가 막히는 세 곳

1. **환경변수 빼먹기** — `.env`는 깃에 안 올리므로, 배포 플랫폼 설정 화면에 비밀값을 따로 넣어야 함. ("로컬은 되는데 배포하니 안 돼"의 최다 원인)
2. **CORS 막힘** — 화면(예: Vercel)과 서버(예: Railway) 주소가 다르면, 서버가 "내 프론트 주소 요청 허용(CORS)"을 열어줘야 함.
3. **무료의 함정** — 무료 티어는 트래픽 늘면 유료로 넘어감.

**셋으로 나눠 배포** (식당 비유):

| 무엇 | 어디에(예) | 식당 | 왜 |
|---|---|---|---|
| 화면(프론트) | Vercel·Netlify | 여러 지점의 홀 | 손님 가까이서 빠르게 |
| 서버(백엔드) | Render·Railway | 늘 켜진 주방 | 요청을 계속 처리 |
| 데이터(DB) | Supabase 등 | 재료 창고 | 안전하게 보관 |

## 9. 집 서버 & 터널

집 기기는 사설 IP + NAT + 유동 IP라 밖에서 곧장 못 들어온다. 원래는 **포트 포워딩 + 동적 DNS(DDNS)**가 필요했지만, 요즘은 **터널**이 편하다 — *밖에서 들어오게 여는 대신, 내 컴퓨터가 바깥 서비스로 먼저 연결을 맺어* 통로로 손님을 받는다(포트 개방 불필요, CGNAT 통과).

| 도구 | 특징 | 언제 |
|---|---|---|
| Cloudflare Tunnel | 포트 개방 불필요, 대역폭 무제한·무료 | 상시 서비스 |
| ngrok | 한 줄로 즉시 공개 주소 | 잠깐 시연 |
| Tailscale Funnel | 사설망으로 묶고 일부만 공개 | 개인·팀 내부 |

## 10. 프론트-백엔드 연결, 초보가 겪는 문제

- **계약 불일치(최대 원흉)** — 프론트가 기대하는 데이터 모양 ≠ 백엔드가 주는 모양. AI를 쓰면 양쪽에 '얼추 맞는' 걸 따로 만들어 더 어긋난다. → 주고받을 데이터를 **처음부터 문서(API 명세)로** 적고 그 기준으로 양쪽 작성.
- **로딩·에러 상태 누락** — 모든 호출을 **로딩 / 성공 / 에러** 세 상태로 나눠 화면 준비.
- **CORS·주소·환경변수** — 배포에서 본 그 문제들이 곧 연결 문제이기도 하다.
